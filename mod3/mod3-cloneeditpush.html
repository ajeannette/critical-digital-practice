Using GitHub

Now that we've set ourselves up, let's go ahead and start using GitHub!

Remember, you're not doing your actual work in GitHub, you're using GitHub to store, track, and share that work with your collaborators (or audience). So, the most important part of working with GitHub is configuring everything appropriately on your own computer, so that that communication is seamless.

There are a few basic practices that you'll need to get a handle on when using GitHub to work on a project, which we'll refer to in this chapter as clone, edit, and push. Cloning and pushing are two terms straight out of the GitHub lexicon, and we'll explain them in detail below. Editing, however, is a general term we'll use to describe the actual work you're doing on your project.

Clone

While not the only way to share a repository between the GitHub website and your machine, we recommend that you start out by "cloning" a repository that you create on your GitHub account in your browser to your machine to initialize communication and tracking.

Create a new repository

We'll begin by going to your GitHub account and clicking on the plus sign next to your profile icon in the upper righthand side of the page. From the dropdown, select "New repository" as shown in the screenshot below.

On the next page, you'll need to name your repository something short but descriptive. The name of your project is a good idea.

Unless you have a GitHub Pro account, you'll have to accept the repository as Public, which means that anyone on the internet can see and submit future pull requests (to be explained later) to your repository. This default somewhat convenient for or purposes, as we are firm believers in public-facing, open work. We're a learning community, which, at the risk of sounding diffuse, means we find community in our efforts to learn, and those efforts, necessarily, should be out in the open. While it can feel vulnerable to work this way, it's the simplest way to share what you're learning with others who may be interested in similar pursuits.

Under "initialize this repository with:" be sure to check "Add a README file" to populate your repository with something. README files act as homepages for repositories, typically used for telling visitors what the project is about, providing essential prerequisite information, pointing to necessary resources, etc. We'll write in the README later.

At this point, click the "Create repository" button.

Cloning to our machine

Now that you have a fresh, new repository you'll have to move it to your computer before you can start adding files to it. To do that, you'll need to click the "Code" button in the upper righthand corner of the file menu. Make sure the "SSH" option is selected underneath the "Clone" header and then click on the copy button to the right of the code line. See the screenshot below:

Now that the SSH link is in your clipboard, go back to your computer, open your terminal, and follow these next steps:

1. cd into where you want your repository to be.

2. Enter git clone then paste in the link in your clipboard, or type git@github.com:youraccount/yourrepo.git 

3. Enter ls to see if your repository is present.

4. cd into your repository and ls to see if your README file is present.

Edit

As mentioned in this chapter's introduction, we're using the term "edit" to refer to the work that you do in your repository from your computer. While you're doing your edits, you won't necessarily be actively interacting with Git, but everything that you move in or out, or change within your repository will ping with Git. 

Let's practice editing by making some changes to that README file.

While you're in your repository directory in your terminal, enter

code README.md

to open the Markdown file in VS Code. You should have this command set up in your command line by now, but, if you don't or can't, simply replace code with open.

Now you can type something in this file. Whatever you want. Then make sure you save it.

Push

Woo, you did it! You made an edit in Git! You may want to rush to your repository on GitHub to check and see if anything has changed, but hold on. There are a couple of things we have to do to before those changes you made can be reflected in the web.

In order to "push" to GitHub (getting your edits on the web version of your repository), you'll need to follow the "add," "commit," "push" workflow.

1. Check what has been changed on your computer by entering

git status

You should see a brief report that lists "Changes not staged for commit" and labels your README as "modified."

2. Stage your changes for commit by "adding" the file:

git add yourfilename.md

or, if there are multiple files, add them all with

git add .

Once you hit enter, if you were successful, you won't see a result, just the command line prompt again. Feel free at this point to go another git status and you should see "Changes to be committed:" with your modified file(s) listed. Now you're ready to commit!

3. To commit, enter the following:

git commit -m "comment describing what you're doing"

Now, this is important. You must include the -m tag and a comment in double-quotes in order to complete your commit. If you don't, you'll be forced to enter a comment in a native CLI text editor like vim or nano, and that's a scenario you'd like to avoid, trust us.

4. Now that you've committed, it's time to push. To do so, enter

git push origin main

Now you can go back to the web version of your repository and refresh. You should see your updated README.
